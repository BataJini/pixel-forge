## Pass 1 — Spec conformance (criteria → met/unmet)

Machine-checkable (`project.acceptance.test.ts`, 4/4):
- Lossless multi-layer / multi-frame round-trip — **MET** (`serialize`/`deserialize`).
- Compact pixel encoding, not a raw JSON number array — **MET** (hand-rolled Base64 of
  the raw RGBA `Uint8ClampedArray`; the `(,0){40,}` compactness assertion passes).
- Malformed / wrong-schema rejection with a result envelope, no throw — **MET**.
- Rejects a project exceeding the 512×512 cap — **MET** (`validateCanvasDim` 1..512).

Manual / review criteria:
1. **Save→reload restores the exact project (autosave + explicit save); debounced
   autosave never loses work** — **MET**. `DocumentStore` (debounced autosave, dirty
   state machine) + `ProjectStore` (IndexedDB via `idb-keyval`). Save-failure keeps the
   live buffer and surfaces an error (unit-verified); autosave restore verified.
2. **Gallery: list w/ thumbnails; open / rename / duplicate / delete; New via Welcome
   (palette + size)** — **MET**. `GalleryDialog` (inline rename, two-step delete confirm,
   duplicate, open, live thumbnails) + `WelcomeDialog` (presets/hardware/custom size,
   built-in palette or free color).
3. **Resize / Crop-to-selection / Trim dialogs modify the canvas correctly and are
   undoable** — **MET (with one deferral, see F-3)**. `ResizeDialog` (W×H + 9-way anchor),
   `CropDialog` (X/Y/W/H), Trim action — all routed through `LayerStack.resizeCanvas` as a
   single undo entry, with extra frames transformed in lockstep. Crop-**to-selection**
   (marquee-driven) is deferred to U-012 where the selection becomes app-global.
4. **Import PNG into a new canvas or as a new layer; oversize (> 512) rejected with a
   friendly error and NO loss of current work; resize/new dialogs clamp/refuse > 512** —
   **MET**. `decodeImageFile` enforces byte + 1..512 dimension caps before allocation and
   returns the envelope; `ImportImageControl` leaves current work untouched on error;
   Welcome/Resize inputs clamp to `MAX_CANVAS`.
5. **Storage-full handled with a visible, actionable message; `navigator.storage.persist()`
   requested** — **MET**. `isQuotaError` → `STORAGE_FULL` ("Export a project… or delete
   saved projects"); `requestPersistentStorage()` invoked once on provider mount.

## Pass 2 — Engineering

Strengths: `deserialize` is a proper defensive boundary (schema gate, 512 cap, exact
`w*h*4` Base64 length check, channel clamps, never throws); the Base64 codec is pure and
DOM-free (keeps `src/core` deterministic); canvas ops are immutable (new buffers, inputs
untouched); the Result envelope is used consistently; the persistence layer is store-
injectable so gallery/quota logic is unit-testable in Node while the real IndexedDB binding
is browser-tested. No over-mocking of the code under test — round-trips run against the real
`serialize`/`deserialize`; `FakeStore`/`FakePersistence` stand in only for IndexedDB, which
is separately covered by browser tests.

### Findings (all advisory — none block)

- **F-1 (LOW) — `openProject` leaves a pending autosave that fires post-load.**
  `openProject` calls `stack.reset()`, whose emit reaches `onDocChange` and schedules an
  autosave before `setSaveState('saved')` runs. ~800 ms after opening a *gallery* project
  with no edit, `runAutosave` bumps `updatedAt` and rewrites the autosave slot. No data
  loss (it writes, never clears), but it is a spurious write + timestamp bump on a clean
  load. *Fix:* after a gallery `openProject`/`openById`, cancel the pending autosave (clear
  `autosaveHandle`) since the just-loaded state is clean; keep the schedule only on
  `restoreAutosave`/`newProject`/edits. `src/state/documentStore.ts:244`.

- **F-2 (LOW) — ineffective lazy import in `projectFile.ts`.** `await import('browser-fs-access')`
  cannot split into its own chunk because `src/platform/exporters/save.ts` statically imports
  the same module (Vite warns at build). The "keeps the dependency off the initial parse
  path" comment is misleading. *Fix:* import `fileOpen` statically (it is already in the
  main chunk) and drop the comment, or make `save.ts` lazy too. Cosmetic; no runtime impact.
  `src/platform/projectFile.ts:45`.

- **F-3 (MEDIUM, advisory → U-012) — Crop-to-selection not wired to the marquee.** The Crop
  op is correct and undoable, but driven by explicit X/Y/W/H rather than the selection
  marquee, because the selection lives in `toolSession` and is not yet app-global (same
  root cause as the documented U-004/U-007 selection/history unification deferrals). *Fix:*
  wire "Crop to selection" to the unified selection when U-012 assembles the shell; add a
  QA/e2e assertion that a marquee crop matches the selection bounds. `src/ui/project/CropDialog.tsx:20`.

- **F-4 (LOW → U-012/U-013) — Unicode arrow glyphs, not bespoke pixel glyphs.** The Resize
  anchor grid (`↖↑↗…`) and the stage undo/redo (`↶↷`) use font arrows instead of the
  bespoke on-grid pixel icons the design-direction calls for. Matches the existing U-007
  L-1 glyph advisory. *Fix:* swap to the U-012/U-013 bespoke icon sprite.
  `src/ui/project/ResizeDialog.tsx:11`.

- **F-5 (LOW) — `deserialize` silently truncates over-long strings.** `asString` caps
  id/name/blend/palette strings at 200 chars instead of round-tripping them, so a
  hand-authored `.forge` with a >200-char name would not be byte-lossless. Harmless for
  app-produced files (title is `maxLength=64`) and is a reasonable defensive normalization
  of untrusted input; noting for completeness. `src/core/project.ts:147`.
