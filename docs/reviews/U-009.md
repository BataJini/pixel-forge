## Pass 1 — Spec conformance (acceptance criteria → status)

Work-breakdown U-009 acceptance criteria:
- **PNG dims = art×scale, no intermediate colors (palette preserved)** — MET.
  `scaleBufferNearest` replicates each source pixel into an exact `scale×scale`
  block (pure, new buffer). Node test asserts every out-pixel equals its source
  block and `colorSet(out) === colorSet(src)` for scales 1/2/4/8; Browser Mode
  repeats it against a real `CanvasRenderingContext2D` and decodes the PNG blob.
- **512×512 @ 4× = 2048×2048** — MET (asserted in both node + browser suites).
- **SVG parses, has viewBox + crispEdges, merged rect count << pixel count,
  re-rasterizes to source** — MET. `bufferToSvg` emits
  `viewBox="0 0 w h" shape-rendering="crispEdges"`, greedy row-run + vertical
  merge (solid 8×8 → 1 rect), omits fully-transparent pixels, encodes partial
  alpha as `fill-opacity` (round-trips every 8-bit alpha within 1/255). Held-out
  test + `svg.test.ts` round-trip L-shapes and checker patterns exactly.
- **Effect-free at 512×512 (no CRT/glow/checkerboard)** — MET, and **structurally**
  guaranteed: exporters take a composited pixel buffer, never the screen.
  `export.browser.test.tsx` proves the PNG bytes are identical with
  `data-crt="full"` vs `"off"`, and that the transparent source pixel stays
  alpha-0 (no grey checker leak).
- **Download via `browser-fs-access` + blob fallback** — MET. `saveBlob` uses
  `fileSave`; e2e deletes `showSaveFilePicker`/`showOpenFilePicker` to force the
  blob-download path and confirms a real `.png`/`.svg` download fires with the
  right suggested filename.
- **Export dialogs** — MET. `ExportDialog` offers scale 1/2/4/8/16/32,
  transparent vs matte (native color chip), live output-size readout, working/
  done/error status, friendly "nothing to export" guard.

## Pass 2 — Engineering / correctness / security

- **Purity & module boundaries** — Correct. Pure encoders under
  `src/core/exporters/{png,svg}.ts` (no DOM except the single spec-mandated
  `scaleToCanvas` boundary, which delegates pixels to the pure scaler). Browser
  glue in `src/platform/exporters`. UI in `src/ui/export`. Barrels additive.
- **Immutability** — Correct. `scaleBufferNearest`/`flattenOnColor` return new
  buffers; `bufferToSvg` is pure. The only mutation (`Run.used`, dst typed array)
  is on function-local data — within the constitution's scoped exception.
- **Result envelope** — Correct. `saveBlob`/`exportPngFile`/`exportSvgFile`
  return `{ok}|{ok:false,error}`; user-cancel (`AbortError`) maps to `cancelled`,
  not an error; genuine failures return friendly codes and do **not** leak
  internal messages (`save.test.ts` asserts "disk full" never surfaces).
- **Security** — No `eval`, no `innerHTML`; SVG is built only from numeric coords
  + `rgbaToHex` (`#RRGGBB`) so no string injection; filename sanitized
  (`sanitizeFileName` strips path-unsafe chars); no network, no secrets. Nothing
  here warrants a separate security-reviewer escalation (no auth/crypto/import
  parsing in this unit).
- **Determinism** — Colors sorted by packed value, rects by (y,x); output stable
  (`a === b` test).

## Findings (all advisory — none block)

### M-1 (MEDIUM) — No output-dimension guard; 512² at 16×/32× can OOM the tab
`PNG_SCALES` offers 32× and the spec allows a 512×512 canvas. 512×32 = 16384²,
i.e. a ~1 GB scaled `Uint8ClampedArray`, plus the ImageData copy and the
OffscreenCanvas backing store — several GB transient, at/over Chrome's canvas-area
limit. `exportPngFile` wraps the encode in try/catch and returns
`EXPORT_ENCODE_FAILED` ("Try a smaller scale."), so it *fails gracefully* rather
than losing work, but an OOM can crash the tab before the catch runs. Latent today
(the wired preview canvas is 32×32), becomes reachable once U-011/U-012 allow
large custom sizes. **Fix:** clamp the offered scales to those whose output
≤ a safe cap (e.g. disable/annotate scale options where `w*scale`/`h*scale`
exceeds ~8192, or a max-area check) before enabling large canvases. Fold into
U-012 (export dialog wiring) or U-013 (perf).

### M-2 (MEDIUM) — "All frames" PNG export not implemented (deferred)
U-009 scope text and the held-out `criteria.md` QA list mention
"current vs all frames (all → sequence zip)", but the dialog exports only the
single current buffer. This is a **dependency-ordering gap, not a builder defect**:
U-009 depends only on U-003, so no Frame model exists yet, and the crisp
work-breakdown *acceptance criteria* + the machine-checkable held-out test do not
require it (all pass). **Fix:** implement the multi-frame PNG sequence export in
U-010 (which already depends on U-008 frames + U-009) and record the deferral in
the decision log so it is not lost. Recommend the manager reconcile the U-009
scope line accordingly.

### L-1 (LOW) — `resolveMatte()` calls throwing `hexToRgba` outside try/catch
In `ExportDialog.runExport`, the action builds args with `resolveMatte()` →
`hexToRgba(matte)`, which *throws* on invalid hex, and this call sits outside the
`try` that guards the export. Safe in practice (matte always comes from
`<input type="color">`, normalized to `#rrggbb`), but not defensive. **Fix:** use
`tryHexToRgba` or wrap arg-building in the guarded path.

### L-2 (LOW) — Export not yet reading a true multi-layer composite
`getSource={() => bufferRef.current}` returns U-003's single art buffer. Correct
for this unit (layers arrive in U-007), and the platform barrel already documents
that the caller must flatten via `core.composite`. Just ensure U-012 wires
`composite(activeFrame.layers)` into `getSource` so the clean-export invariant
holds once layers exist. No change needed in U-009.
