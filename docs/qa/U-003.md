**Live drag stroke** (60 pointer moves at 1600% on the real app), frame cadence via
instrumented `requestAnimationFrame`:
- avg **16.66 ms**, p50 16.70 ms, p95 **16.70 ms**, max **16.70 ms** → steady

  "a single pointer op … within one animation frame".

**Pure hot path at the 512×512 ceiling** (micro-bench via Vitest):
- `setPixelMut`: **11.4 ns/op** (draw hot path — negligible)

  recomposite, **not** per pointer edit (render path uses `updateRegion` with the
  explicit dirty rect). Flagged for U-007/U-008, which already require frame caching.

## 4. Findings (all non-blocking)

### F-1 (LOW) — Wheel-zoom does not update the status-bar zoom % readout
`src/ui/CanvasStage.tsx` `onWheel` (~L107–113) calls `renderer.setViewport(...)`
but, unlike the toolbar button handlers, never calls `setZoomPct(...)`. **Verified
root cause:** wheel events reach the handler (`defaultPrevented === true`) and the
canvas **does** zoom (opaque display px 154→616 = exact cursor-centered 2× step),
yet the "Zoom NN%" indicator stays stale (e.g. 100%) until the next button
interaction. Engine zoom is correct; only the preview's readout desyncs.
*Fix:* in `onWheel`, after `setViewport`, `setZoomPct(Math.round(next.zoom * 100))`
— or derive the displayed zoom from a single renderer-viewport source of truth.
Not covered by any acceptance criterion (the criteria govern coordinate mapping,
which is correct) and confined to preview scaffolding replaced by U-012.

### F-2 (LOW / advisory) — Fractional pan produces a 1px AA seam on the checker backdrop
`fitToScreen` can return a fractional pan (observed `panY = (681−512)/2 = 84.5`).
The checkerboard is a vector `fillRect` drawn at that fractional offset, so cell
edges anti-alias into a ~1px softened seam (sampled `[148,148,148,234]` at a cell
boundary). **The artwork itself stays perfectly crisp** — the display canvas has
**0 partial-alpha pixels** at that view and shows only exact art colors, and the
**clean-export invariant is intact** (backdrop is never in the buffer/export).
*Fix (polish):* snap pan to integers (`Math.round` on `panX/panY` in `fitToScreen`
and/or on `setViewport`).

### F-3 (advisory / scope) — Space+drag and two-finger/pinch pan not wired in the preview
The held-out `criteria.md` manual list anticipates "pan via Space+drag, middle-drag,
and two-finger drag." The U-003 preview wires **middle/right-drag pan + cursor-
centered wheel zoom**; Space+drag and pinch/two-finger are **U-012 scope**
(app-shell input map + touch, per `work-breakdown.md`). The engine already exposes
everything needed (`panBy`, `zoomAt`, `screenToArt`); only the bindings are deferred.
The authoritative U-003 acceptance criteria ("zoom/pan map coordinates correctly")
are met.
