## Findings

No blocking or high findings. The prior blocker is resolved.

### Resolved — prior F-1 (HIGH, BLOCKING): Layers panel keyboard-inoperable — FIXED
- **Prior defect:** `CanvasStage.tsx`'s window-level `keydown` handler `preventDefault()`ed
  Enter/Space for every non-`INPUT/TEXTAREA/SELECT` target, cancelling native activation of
  every panel `<button>` (keyboard-only users could not manage layers at all).
- **Fix (verified in `-bea-6`, `CanvasStage.tsx`):** a new `isInteractiveTarget` guard returns
  early from the handler — *before* the Enter/Space `preventDefault` — when
  `e.target.closest('button, [role="button"], a[href], input, select, textarea, [tabindex]')`
  matches. Modifier combos (Ctrl/Cmd+Z/Y/A/C/X/V) are handled earlier and remain global.
- **Live re-verification:** Enter on focused Add adds a layer with `defaultPrevented=false`;
  Space on a focused eye toggles visibility with `defaultPrevented=false`; keyboard-only
  undo/redo of layer ops works via the panel's own Revert/Reapply buttons. Covered by two new
  E2E cases (`layers.spec.ts:71,79`) and a browser test (`LayersKeyboard.browser.test.tsx`),
  both green.
- **No regression:** canvas drawing shortcuts still fire (the canvas well has no `tabindex`, so
  its target never matches the interactive selector); E2E history (Ctrl+Z/Y, drag=1 undo) and
  smoke (zero console errors) pass. Bundle/coverage/lint unchanged.

### Advisory / non-blocking
- **"Ctrl+Z does not undo a layer add" — expected, NOT a defect (harness confound).** On this
  shared preview page, `Ctrl+Z` (`CanvasStage.tsx:220`) is wired to the **canvas** ToolSession
  history; the Layers panel intentionally keeps its **own** Revert/Reapply timeline (code
  comment: "…never clash with the canvas toolbar's Undo/Redo… U-012 unifies them into one
  timeline"). The U-007 requirement — every layer op undoable, keyboard-only — is met via the
  panel's own buttons (verified keyboard-operable). Unifying the two timelines onto one global
  Ctrl+Z is explicitly **U-012** scope. My driver's one non-pass was a mis-scoped expectation.
- **Security smoke — PASS.** A layer name of `<img src=x onerror="window.__xss=1">` renders as
  inert text: `window.__xss` stays `undefined`, no `<img>` injected, value stored verbatim
  (React text binding; no `dangerouslySetInnerHTML`). No backend/secrets. Zero console/page
  errors across the whole session and on load.
- **A11y (pointer/AT) positives.** Two labelled `role="toolbar"` groups; `aria-current` on the
  single active row; `aria-pressed` on visibility/lock toggles; labelled opacity slider and
  layer list; eye/lock state read by SHAPE + `aria-pressed`, not hue alone; a visible focus
  target confirmed inside the panel.
- **Reward-hack posture — clean.** `git diff master -- docs/acceptance/U-007/` is empty
  (held-out unchanged); `vite.config.ts` adds only the `docs/acceptance/U-007/**` include line
  (legit activation per config policy); no `.skip/.only/.todo/xit/xdescribe/@ts-ignore/
  @ts-nocheck` in any new test file; mutation probe proves the suite binds to the impl.
