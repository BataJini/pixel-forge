## Machine-checkable criteria (exercised in a real browser)

### 1. Command palette (Ctrl/Cmd+K) opens, fuzzy-searches, runs the same action as the menu ‚Äî **PASS**
- `Ctrl+K` opens `.pf-cmdk` (`role="dialog"`, `aria-modal="true"`, search input auto-focused);
  **22 commands** render (disabled Undo/Redo/Copy/Cut correctly filtered out).
- Typing `export` ranks **File ‚Üí Export‚Ä¶** first (`firstItem = "File Export‚Ä¶"`).
- `Enter` closes the palette (`paletteClosed=true`) and opens `dialog.pf-export` (`open=true`,
  heading **"Export"**) ‚Äî identical to the "Export‚Ä¶" menu/toolbar action.
- Evidence: `qa2-02-export-via-palette.png`; capture in `qa2-results.json ‚Üí crit1_palette_export`.

### 2. Representative shortcut set fires the right command ‚Äî **FAIL (Ctrl+S save missing)**
Driven through the app's real `window` keydown handler; tool value read from the status bar's Tool
readout, zoom from the Zoom readout, a `defaultPrevented` probe on the bubble phase, and a real
mouse-drawn stroke for undo.

| Shortcut | Expected | Observed |
|---|---|---|
| B / E / G / L / U / I / M / V (/ H) | pencil / eraser / bucket / line / rect¬∑ellipse / eyedropper / select / move (/ hand) | **all correct** ‚Äî status "Tool ‚Ä¶" updated each time; **U toggles rect‚Üîellipse** (`toolKeys_corrected`, every `ok:true`) |
| `+` / `-` | zoom in / out | **works** ‚Äî 1200% ‚Üí 1600% ‚Üí 1200% (`crit2_zoom`) |
| Space | pan | **works** ‚Äî `defaultPrevented=true` |
| `Ctrl+Z` / `Ctrl+Shift+Z` | undo / redo | **works end-to-end** ‚Äî real stroke ‚Üí "1 step" (undo enabled) ‚Üí `Ctrl+Z` ‚Üí "0 steps" (undo disabled, redo enabled) ‚Üí `Ctrl+Shift+Z` ‚Üí "1 step" (`undo_redo_corrected`) |
| **`Ctrl+S`** | **save** | **NOT handled** ‚Äî probe shows **`defaultPrevented=false`**; no `file.save` command (palette "save" ‚Üí empty state); no File ‚ñ∏ Save item. `Ctrl+N` / `Ctrl+O` likewise **`false`** (unbound). Control-check: `Ctrl+Z`/`Ctrl+K`/`Ctrl+A` all read `defaultPrevented=true`, proving the probe is valid. |

The criterion explicitly lists "Ctrl+S save"; it is absent. **Criterion fails.** Root cause:
`attachKeyboard` (`src/ui/CanvasStage.tsx:215‚Äì302`) handles Ctrl+Z/Y/A/D/C/X/V then bails with
`if (mod || e.altKey) return;` ‚Äî every other modified key falls through unhandled; the separate
shell effect (`CanvasStage.tsx:528`) handles only Ctrl+K / `?` / `¬±`. Evidence:
`qa2-micro-results.json ‚Üí defaultPrevented_exact`.

### 3. Menus File/Edit/View/**Canvas**/Help each open; every item invokes a wired command ‚Äî **FAIL (no Canvas menu)**
Live menu bar: **File ¬∑ Edit ¬∑ View ¬∑ Tools ¬∑ Help** (`hasCanvas=false`). No command carries
`group:'Canvas'`, so `MenuBar` never renders it (`MENU_ORDER.filter(g => commands.some(c => c.group===g))`,
`MenuBar.tsx:41`). Every present menu was opened and enumerated ‚Äî **all items wired, no dead items**
(contextually-disabled items behave correctly):
- **File** (2): Export‚Ä¶, Clear Canvas
- **Edit** (8): Undo\*, Redo\*, Select All, Deselect, Copy\*, Cut\*, Paste, Swap Colors  (\*disabled without history/selection)
- **View** (4): Zoom In, Zoom Out, Reset Zoom (100%), Fit to Screen
- **Tools** (10): Pencil ‚Ä¶ Pan
- **Help** (2): Keyboard Shortcuts, Command Palette

Menu‚Üícommand wiring verified live: **View ‚ñ∏ Reset Zoom (100%)** changed the status bar **1200% ‚Üí 100%**
(`menu_wiring_resetZoom`). But master-spec ¬ß2/¬ß3.7 mandate a **Canvas** menu (Resize, Crop to
selection, Trim transparent, Change palette, Indexed mode) and a fuller **File** menu (New, Open,
Save, Save As, Import Image, Export ‚ñ∏, Duplicate, Delete). None of the Canvas commands exist in the
shell, and File is missing New/Open/Save/etc. **Criterion fails on the Canvas menu.**
(Notably the underlying actions *do* exist ‚Äî the U-011 ProjectWorkbench panel exposes its own
New / Open / Save / Resize / Crop / Trim / Import buttons ‚Äî but they are **not surfaced through the
U-012 menu bar, command palette, or Ctrl+S** as the criteria require.)

### 4. First run shows Welcome/New onboarding; Help, Settings, Gallery overlays open & close ‚Äî **FAIL (2 of 4 clauses)**
- **First-run Welcome onboarding ‚Äî FAIL.** In a **brand-new, never-visited browser context**, and
  again after `localStorage.clear()` + `sessionStorage.clear()` + deleting IndexedDB, **no dialog
  auto-opens** (`dialog[open]`=0, `welcomeOpen=false`; only `pixelforge.recentColors.v1` is written
  to localStorage ‚Äî no first-run flag). The Welcome/New-Canvas UI **exists** and opens via the "New"
  button (`dialog.pf-welcome`, heading **"New Canvas"**, size presets 8¬≤‚Äì128¬≤, palettes Game Boy /
  NES tile, "Forge it") ‚Äî it just never auto-appears for a new user. `App.tsx:56` mounts
  `<ProjectWorkbench/>` with **no `autoWelcome`** (defaults `false`), so `firstRun` never becomes
  true. Evidence: `qa2-01-firstrun-cleared.png`, `qa2-05-welcome-via-new.png`;
  `qa2-focus-results.json ‚Üí freshContext_firstRun`, `qa2-micro-results.json ‚Üí new_opens_welcome`.
- **Settings overlay ‚Äî FAIL (does not exist).** Palette searches `settings` and `theme` both
  return the empty state; **0** settings/preferences DOM nodes anywhere; no Settings command in any
  menu (`crit4_settings`, `crit_palette_probes`). Theme / CRT / sound / grid defaults (master-spec ¬ß2
  `#settings`) are unreachable in the running app.
- **Help overlay ‚Äî PASS.** Help ‚ñ∏ Keyboard Shortcuts opens the overlay (**19 shortcut rows**) and
  **closes on Escape** (`crit4_help`, `closesOnEscape=true`). Evidence: `qa2-03-help-overlay.png`.
- **Gallery overlay ‚Äî PARTIAL.** Opens via the Project panel "Open" button (`dialog.pf-gallery`),

## Manual / review criteria

- **Full workbench layout ‚Äî mostly present, deviates from spec.** Menu bar ‚úì; center canvas well
  with seeded anvil motif ‚úì; right dockable panels (Project / Color-Palette / Layers / Frames) ‚úì;
  bottom status bar ‚úì (Pos / Color / Zoom / Size / Tool / Sel ‚Äî raw:
  `Pos 0,0 Color #00000000 Zoom 1200% Size 32√ó32 Tool pencil Sel ‚Äî`). But the "left tool rack" is a
  set of **horizontal toolbars**, not the mandated **vertical bevelled rack** (master-spec ¬ß3 "Left
  tool rack (vertical)"), and the surface is the stitched-together per-unit demos rather than the
  unified three-zone forge. Evidence: `qa2-07-desktop-workbench.png`.
- **Disjoint state ‚Äî confirmed live.** The workbench mounts multiple independent surfaces, each with
  its OWN history: the main canvas ("Edit history": Undo/Redo/steps buttons) and the Layers panel
  (its own `‚Ü∂`/`‚Ü∑` revert/reapply) are both present in the live button census. `App.tsx`'s own doc
  comment states ProjectWorkbench "keeps its OWN layer stack so the pre-existing tool/color/layers
  demos above stay untouched." Global `Ctrl+Z` drives only the main Edit history; main-canvas
  strokes do not land on the Layers panel's active layer. The single-session / single-timeline forge
  master-spec ¬ß2/¬ß3 describes is not assembled here.
- **Responsive / touch [Q2] ‚Äî NOT MET.** At a 375 px viewport the dock is a plain
  `flex-direction:column` full-width stack (`bottomSheetNodes=0`; no page-level horizontal scroll,
  `scrollW==clientW==375`) ‚Äî not the bottom-sheet system the intake Q2 scope lists. The pointer
  handler (`CanvasStage.tsx` `attachPointer`, single `pointerdown`/`pointermove`, no multi-touch
  cache / no distance math) implements **no pinch-zoom and no two-finger pan** (master-spec ¬ß3 Pan:
  "two-finger drag (touch)"). Evidence: `qa2-06-mobile-375.png`, `manual_responsive_375`.
- **Keyboard reachability + visible focus ring ‚Äî PASS.** Tabbing reaches controls; the focused
  control matches `:focus-visible` and shows the bespoke **Spark ring**
  (`box-shadow: rgb(6,7,12) 0 0 0 2px, rgb(255,46,136) 0 0 0 4px ‚Ä¶`) (`manual_focus_ring`). Advisory
  a11y polish (not gated by the criteria): the menu bar is Tab+Enter only (no arrow-key roving) and
  the palette has no focus trap/restore.
