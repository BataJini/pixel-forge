## 4. Findings

### QA-1 (MEDIUM, non-blocking) — Escape does not close U-011 modals in the integrated app; also silently clears the canvas selection
- **Where:** `src/ui/CanvasStage.tsx:239-242` (U-003), interacting with `src/ui/components/Dialog.tsx`.
- **What:** The U-003 `CanvasStage` installs a global `window` `keydown` listener. For
  `e.key === 'Escape'` it calls `e.preventDefault()` and clears the canvas selection
  **before** the interactive-target bail-out (line 266) and unconditionally at the
  window level. Because it cancels the Escape keydown, the browser suppresses the
  native `<dialog>` close-request, so the `cancel` event `Dialog.tsx` relies on
  (`Route the native Esc/cancel through our controlled onClose`) never fires. In the
  shared App (which mounts both `CanvasStage` and `ProjectWorkbench`), **Escape does
  not close** the Welcome / Gallery / Resize / Crop dialogs.
- **Second effect:** every Escape press while a U-011 dialog is open also mutates the
  underlying U-003 canvas selection — a surprising cross-surface side effect.
- **Repro:** Open the app → click **Open** (Gallery) → press **Escape** → dialog stays
  open (evidence `13-escape-did-not-close-gallery.png`; observed `stillVisible=true`
  for Welcome, Gallery, and Resize).
- **Why non-blocking:** not in the U-011 acceptance criteria; not a WCAG AA failure
  (dialogs are fully dismissible via keyboard Close/Cancel and via backdrop, both
  verified); no data loss. In U-011's own component tests `CanvasStage` isn't mounted,
  so the unit tests pass — this only manifests in the integrated shell.
- **Fix hint:** In the `CanvasStage` global handler, ignore Escape (and ideally all
  keys) while a modal dialog is open, e.g. bail early when
  `document.querySelector('dialog[open]')` exists or when
  `e.target`/`activeElement` is inside a `dialog`. That restores Escape-to-close and
  removes the selection-clear side effect. Natural to resolve in **U-012**, which
  unifies these shells and owns global shortcut routing. Recommend a regression E2E:
  "Escape closes each project dialog and does not disturb the canvas."

### Minor observations (not defects)
- **Sub-debounce reload window:** autosave is debounced at 800ms; a reload within
  800ms of the very last stroke can lose only edits since the previous autosave. This
  is inherent to debouncing and mitigated by frequent autosave + explicit Save +
  Export `.forge`. The lossless round-trip itself is correct (verified byte-identical).
- **Bundle:** app JS 332.9 kB (**gzip 104 kB**) and CSS 35.3 kB (**gzip 6.5 kB**) — within
  the app-page budget (JS < 300 kB gzip, CSS < 50 kB). Canvas ops are pure typed-array
  loops bounded by the 512² cap; no perf concern observed.
- **Security smoke:** fully client-side, no secrets. `deserialize` is a defensive
  boundary parser (schema check, 1..512 cap, per-field validation, channel clamping,
  Base64 length check) that never throws on untrusted input. Imported images are
  decoded via a 2D canvas (not `innerHTML`); thumbnails are self-generated PNG data
  URLs. Storage-full is caught and reported without leaking internals. No XSS/injection
  vectors surfaced.
- The shared preview App mounts `ProjectWorkbench` with `autoWelcome={false}` by design
  (so the U-004/U-007 demos on the same page stay interactive); the mandatory first-run
  Welcome is enabled by the standalone shell in U-012. Users still reach New via the
  "New" button — verified.
