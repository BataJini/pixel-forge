## 3. Live end-to-end exercise (own Chromium, CRT forced FULL)

Driven through the real Export dialog with `document.documentElement`'s
`data-crt="full"` (on-screen scanlines visible — see `01-export-dialog.png`), the
exported bytes were captured at the `browser-fs-access` blob boundary and decoded:

- **PNG 8× transparent:** valid PNG signature, 256×256, `corner(0,0)=[0,0,0,0]`
  (no checkerboard leak), seeded spark `px(128,88)=[255,224,138,255]` and body
  `px(96,128)=[58,52,46,255]` **exact**, **6 distinct colors** (nearest-neighbor
  introduces none), status "PNG exported." → `02-export-8x-transparent.png`
- **PNG 8× matte (#12100E):** 256×256, `corner(0,0)=[18,16,14,255]` (flattened,
  fully opaque), art pixels unchanged → `03-export-8x-matte.png`
- **SVG:** `image/svg+xml`, parses clean, `viewBox="0 0 32 32"`,
  `shape-rendering="crispEdges"`, width/height 32, **8 rects / 5 color groups**
  (merged; far below the ~pixel count) → `05-export.svg`

This is the **clean-export invariant** (constitution) demonstrated at the real UI
layer, independent of the browser-mode unit test.

## Findings

### F1 (LOW / informational, non-blocking) — "All frames" PNG export not offered
`master-spec §3.8` and the U-009 work-breakdown scope list "current frame vs all
frames (all → sequence zip)", and `docs/acceptance/U-009/criteria.md` repeats
"current vs all frames" as a manual QA item. The delivered `ExportDialog` exports
the **current frame only** — there is no "all frames" control.

- **Why non-blocking:** the frame/timeline model is **U-008 (pending)**, and U-009's
  declared dependency is **only U-003** — so multi-frame export is not implementable
  at this position in the graph. "All → sequence zip" also overlaps **U-010**
  (spritesheet / multi-artifact), which *does* depend on U-008. The single-frame app
  has exactly one buffer today, so current-frame export is the only coherent
  behavior. The **authoritative machine-checkable** held-out acceptance does not
  require frames, and it passes.
- **Fix / follow-up:** treat "all frames → sequence zip" as U-010 scope (or a U-009
  follow-up gated on U-008). Recommend reconciling the docs: either drop the
  "all frames" line from U-009's scope/criteria or add a `U-008` dependency. Re-verify
  multi-frame export when U-008/U-010 land.

No other defects found. Every machine-checkable acceptance item, the clean-export
invariant, and all relevant test types (smoke, functional, regression, integration,
a11y, performance, security smoke) pass on real runs.
